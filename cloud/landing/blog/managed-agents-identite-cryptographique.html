Claude Managed Agents : pourquoi les agents en production ont besoin d'identite cryptographique

9 avril 2026 · AIB Team

managed-agents anthropic identite-agent EU AI Act MCP

Anthropic vient de lancer Claude Managed Agents en beta publique. Ce service permet aux entreprises de deployer des agents IA autonomes sur l'infrastructure d'Anthropic, sans gerer le provisioning, la gestion d'etat ou la recuperation d'erreurs. Notion, Rakuten, Asana et Sentry font partie des premiers utilisateurs.

C'est une etape majeure pour l'ecosysteme agentique. Mais cette avancee souleve une question que personne ne pose encore assez fort : comment identifier, verifier et auditer ces agents une fois en production ?

Ce que Managed Agents resout

Jusqu'ici, deployer un agent en production necessitait des mois de travail d'infrastructure : sandbox securise, gestion de contexte, orchestration d'outils, permissions, reprise sur erreur. Managed Agents encapsule tout cela dans une API.

Le service propose des sessions persistantes, du checkpointing, des permissions scopees et du tracing. Les agents peuvent executer du code dans un sandbox, appeler des outils via MCP, et meme (en preview) orchestrer d'autres agents.

Le pricing est transparent : tokens Claude standard + 0.08$/heure de runtime actif.

Ce que Managed Agents ne resout pas

Managed Agents est une plateforme d'execution. Elle repond au "comment faire tourner un agent". Mais elle ne repond pas aux questions suivantes :

  • Identite portable — Comment un agent prouve-t-il son identite aupres d'un autre agent, d'un service tiers, ou d'un humain, independamment de la plateforme qui l'heberge ?
  • Interoperabilite des credentials — Comment un agent Claude communique-t-il de maniere verifiable avec un agent GPT, Gemini, ou un agent custom via des protocoles differents (MCP, A2A, AG-UI) ?
  • Audit cryptographique — Comment prouver a posteriori qu'une action a ete realisee par un agent specifique, avec des capacites specifiques, a un moment precis ?
  • Conformite reglementaire — Comment satisfaire les exigences de l'EU AI Act (Articles 9, 12, 13, 14) en matiere de tracabilite, transparence et supervision humaine des systemes d'IA a haut risque ?
Le parallele historique : quand les conteneurs Docker ont democratise le deploiement d'applications, personne ne s'est demande "qui est ce conteneur ?" pendant les deux premieres annees. Puis sont arrivees les failles de supply chain, et soudain l'identite, la signature et la verification sont devenues critiques. Les agents IA suivent exactement la meme trajectoire — en accelere.

La couche manquante : l'identite cryptographique des agents

C'est exactement le probleme qu'adresse le protocole AIB (Agent Identity Bridge). AIB n'est pas une plateforme d'execution concurrente de Managed Agents — c'est la couche d'identite qui se branche dessus.

Concretement, AIB fournit :

  • Des passeports cryptographiques pour chaque agent (cles Ed25519, capacites declarees, protocoles supportes)
  • Des receipts signes pour chaque operation — preuves cryptographiques immuables et verifiables
  • La traduction de credentials entre MCP, A2A, ANP et AG-UI — pour que les agents communiquent entre plateformes
  • Des policy rules (rate limits, separation of duties, capability gates) integrables comme guardrails
  • La conformite EU AI Act by design — tracabilite, transparence et supervision integrees au protocole

Comment AIB s'integre avec Managed Agents

Managed Agents supporte MCP pour l'orchestration d'outils. AIB expose un MCP Server qui permet a tout agent manage d'obtenir et verifier des identites cryptographiques directement depuis son flux d'execution.

Un agent Claude heberge sur Managed Agents peut ainsi :

  1. S'authentifier avec un passeport AIB avant d'interagir avec des services tiers
  2. Generer des receipts signes pour chaque action critique
  3. Verifier l'identite d'un agent distant avant de lui faire confiance
  4. Appliquer des policies de gouvernance definies par l'organisation

L'integration est naturelle : pas de refonte, pas de migration. AIB s'ajoute comme un outil MCP supplementaire dans la configuration de l'agent.

Pourquoi c'est urgent

Le lancement de Managed Agents va accelerer massivement le nombre d'agents en production. Anthropic n'est pas seul — Salesforce avec AgentForce, Microsoft avec Copilot Studio, Google avec Gemini Cloud poussent dans la meme direction.

Chaque plateforme construit son propre silo d'identite. Les "scoped permissions" de Managed Agents ne fonctionnent que dans l'ecosysteme Claude. Les credentials AgentForce ne sont pas reconnus par Copilot Studio.

Il manque une couche d'identite ouverte, portable et verifiable. C'est exactement le role d'un protocole comme AIB — neutre vis-a-vis des plateformes, interoperable par design, et auditable cryptographiquement.

Ce que cela signifie pour les entreprises

Si vous deployer des agents en production — que ce soit sur Managed Agents, sur votre propre infrastructure, ou en multi-cloud — posez-vous ces questions :

  • Pouvez-vous prouver quel agent a fait quoi, et quand ?
  • Vos agents peuvent-ils s'identifier aupres de partenaires externes ?
  • Etes-vous conforme aux exigences de tracabilite de l'EU AI Act ?
  • Votre identite agent est-elle portable, ou prisonniere d'un vendor ?

Si la reponse a l'une de ces questions est "non" ou "je ne sais pas", il est temps d'explorer une couche d'identite cryptographique pour vos agents.
